## D-020 — Audio: ONE smconv pipeline; SFX are IT samples; modules are synthesized by mkit.py
**2026-07-13** · Context: the Phase 3.5 audio spike (R3 — the last unproven pipeline). Decision: (a) **one soundbank pipeline**: `tools/audio/mkit.py` deterministically synthesizes two Impulse Tracker modules (no binary assets, the mktiles pattern) → `smconv -s -b 5` → a dedicated ROM bank 5; ARCHITECTURE's imagined second path (`assets/sfx/*.wav → snesbrr`) is dropped — snesmod's BRR-file mode (`spcPlaySound`) STOPS module playback by design, and IT-sample SFX ride the same converter. (b) **SFX = the first IT's six samples** (jump, fire, portal-open, teleport, land, death — ids in sound.h must match mkit.py's sample order), loaded via `spcLoadEffect(0..5)` after every `spcLoad` (a load resets the ARAM layout — the effectsandmusic example's pattern). `track01.it` carries the six SFX as its OWN samples 1-6 too: the example leaves spcLoadEffect's indexing ambiguous (soundbank-first-IT vs currently-loaded-module) and the duplication makes BOTH readings land the same sounds for ~4KB of already-budgeted ARAM — trim once the semantics are proven on hardware. (c) **Voice budget (GDD 6+2)**: music is authored on channels 1-6; snesmod overwrites channel 8 with effects (pvsneslib_snesmod.txt), 7 stays clear. No echo — EDL would eat 2KB ARAM per 16ms. (d) **INV-AUD-001 is enforced at build time**: `tools/audio/checkbank.py` asserts sfx-module + largest-music-module ≤ 58KB (snesmod's documented post-driver budget) on smconv's own generated sizes, and the `.bnk` ≤ its 32KB bank. (e) Boot order: `spcBoot()` is main()'s FIRST statement (NMI still off); the slow soundbank→ARAM load runs BEFORE the debug magic so tests never see a stalled boot; `spcProcess()` runs once per frame inside the dbg_mainv bracket (measured: invisible — idle max 53 scanlines, lag 0). (f) A room load re-grounds the player, which fired a phantom LAND on every transition (probe-caught) — `sfx_quiet(8)` suppresses SFX for 8 frames after audio_start/goto_room. Mix knobs live in sound.c (sfx_volpan/sfx_pitch tables, spcSetModuleVolume) and mkit.py (composition) — they are mix, not gameplay, so NOT tuning.h. Alternatives: snesbrr BRR files + spcSetSoundEntry (rejected: kills music, second pipeline); hand-tracked .it assets (deferred to Phase 6 — the spike proves the pipe; a musician file drops into AUDIOFILES unchanged).

## D-019 — Title + end-of-demo card: console text, release-boot only, verb-testable
**2026-07-13** · Context: Phase 3.5 title screen + end card. Decision: both cards are pvsneslib console text (Phase 0's proven layout: map $6800, gfx $3000, offset $0100, the gfx4snes example font restored to data.asm) drawn over a force-blanked BG1 repoint; the NMI uploads the text while `vq_console` is set (vblank.c chains the lib's `consoleVblank`, which no pvsneslib header declares). The world PARKS behind a card; START dismisses through `goto_room`, which reloads everything the card clobbered (map region + tile 0 + OAM are blanked at card entry — stale room VRAM showed through the first build). **Release boots into the title; TEST builds boot straight to the hall** and summon cards via verb 10 — a boot-blocking title would stall every harness test; the tested path is the same `game_title()` both flavors call. The chamber's puzzle exit now shows the END CARD (the demo's arc), and its START returns beside the pit. Alternatives: a generated logo-tile pipeline (deferred — the console card is Milestone-A-honest, art can replace it later); title-in-TEST-boots with harness START presses (rejected: touches every test for one path).

## D-018 — Death = the door-fade path back to the room entry (full reset)
**2026-07-12/13** · Context: Phase 3.5 death/respawn. Decision: a sentry-shot contact sets `ent_hit_player`; main.c rides the D-017 fade to the CURRENT room's default entry. Respawn is a FULL room reload — portals recalled, entities respawned — which IS the room-reset guarantee INV-GAME-001 leans on (no partial-state respawn bugs, deterministic per INV-ENG-002). No health/hitstun in v1 (one hit = reset; rooms are small); the chamber has no damage sources yet (the drone gains contact damage only when a chamber respawn story exists). `dbg_death` at +70. Alternatives: respawn-in-place with invulnerability frames (rejected: partial-state bug surface, and the GDD's rooms are checkpoint-scaled); a lives counter (rejected for the demo — friction without stakes).

## D-017 — Doors: UP-to-enter trigger rects + NMI-shadow brightness fades
**2026-07-12** · Context: Phase 3.5 door transitions. Decision: (a) a door = a hand-tabled px trigger rect (main.c `door_tab`, D-016 convention) entered by pressing **UP while standing in it** (R excluded — R+UP is aim-lock). Up-gating is load-bearing: auto-triggers proved hostile to scripted tests (test_room walks Wren across the whole floor and can land in the pit; any floor-level rect near a wall collides with test_walk's wall-pin), and no scripted test presses bare UP — placement becomes unconstrained. (b) The fade is a one-shot **brightness shadow applied inside the NMI** (`vq_set_bright`, vblank.c — $2100 written mid-display tears a scanline); fade-out 2 steps/frame → `goto_room()` (the one force-blank mode/room switch, shared with the TEST warp mailbox) → fade-in. The world freezes during fades. (c) The door VISUAL is a new metatile ('D', tiles 35-38, collision EMPTY) appended AFTER the pinned portal tiles 27-34. (d) The chamber's puzzle recess is now a REAL exit (`cham_exit` → fade back to the hall) — the shaft-escape caveat in JEREMY-INBOX is obsolete once you've solved the puzzle. Doors: hall west ↔ room02 east, the pit floor → the chamber. Note: a transition takes ~40-60 real frames end-to-end (the 8K-word live-map copy + the force-blank redraw dominate, fades are 7 frames each) — acceptable for doors; revisit if it ever gates on feel. Alternatives: auto-trigger overlap doors (rejected: the test-path collisions above + accidental entries); pvsneslib setFadeEffect (rejected: blocking, freezes dbg_frame and pad scripts mid-fade).

## D-016 — Multi-room runtime: ONE live room map in WRAM; rooms share the tileset
**2026-07-12** · Context: Phase 3.5 needs Zone 1's 7 rooms, but every collision/streaming hot path direct-indexed `room01_map` (player, entity, portal validator, and seams.asm's MVN/gather — an asm bank byte). Decision: mirror D-015 — `room_map` is a 16KB WRAM buffer in bank `$7F` (chamram.asm, beside `cham_map`); `room_load(id)` copies the selected `roomNN_map` ROM original into it under force-blank and re-sums the COPY against that room's checksum (INV-ENG-005 now also proves the copy). Every reader and seams.asm index the one label — zero per-word indirection, and rooms become mutable for free (switches, breakable brass later). Attributes: all rooms share ONE tileset, so hot paths keep indexing `room01_att`; roomglue ASSERTS every room's att table is byte-identical at build time (a second tileset must grow a runtime att story first). Room ids: warp id 1 = the chamber (established by tests); other ids map to authored rooms (0→room01, 2→room02, …) via room_load's copy chain — revisit the numbering when Zone 1's map is authored. Spawns and per-room player entry points are hand-tabled by id in ent_room_init/main.c until the Zone 1 authoring pass earns a generated spawn-from-ASCII table. All rooms are the D-012 128×64 grid. Alternatives: per-room extern labels at every read site (rejected: 7× the hot-path code and seams.asm can't switch banks cheaply); stored C pointers to ROM (rejected: the tcc816 bank rule).

## D-015 — The puzzle lives IN chamber01; the chamber map is a mutable WRAM copy
**2026-07-12** · Context: Phase 3's "one complete puzzle chamber (two reorientations + a crate)". Decision: (a) the puzzle is authored INTO chamber01 (the test arena) — pad in the ceiling (metatile row 1 cols 6-7), exit door in the left wall's ceiling corner (col 1 rows 2-3), crate on ledge B — placed so none of the four green chamber tests' load-bearing geometry (faces, brass strips, ledges, spawn, drone leash) moves; a second chamber id arrives with Zone content, not before. (b) `cham_map` became a 16KB WRAM copy in bank `$7F` (`chamram.asm`, copied from `cham_rom_map` under force-blank at load) so an opening door mutates ONE source of truth for all four solid readers (chamber csolid/eject, entity solid_at, portal validator) — the alternative, a second overlay mechanism in every reader, is where parallel-path bugs breed; this also preps magnet-plates (Phase 5). (c) Frame-budget rules learned by scanline probe: the pad probe runs only on a crate-sleep EVENT (`ent_slept`), the door opens one row per frame (a free slide animation), and the gravity-change burial eject SNAPS flush past the binding solid on the first tween frame instead of stepping per-pixel on the transit frame (per-px `craw_box` stepping measured a full frame over budget; flat-burial results are identical). (d) The chamber crate DROP is a straight anti-gravity pop with no tangent impulse (rooms keep the facing throw) — predictable pad placement under any gravity. Crates fall along the live gravity vector and wake on gravity changes. Alternatives: a second chamber + multi-chamber pipeline tonight (rejected: machinery without a design need); door-as-overlay flag in each solid reader (rejected: parallel paths). Follow-up noted: Wren walks THROUGH chamber crates (ent_clamp_* isn't wired in chamber mode) — revisit if a puzzle needs crate-as-platform.

## D-014 — Gale Drone: a LEASHED floater — gravity/portal/solid-free by design
**2026-07-12** · Context: Phase 3's chamber enemy. Decision: the drone bounces its center inside spawn ± `DRONE_R` (tuning.h) with a constant drift velocity — it reads no collision, ignores gravity changes, and never rides portals; authored spawns MUST give it a clear patrol box (chamber01: center 512,500 ± 56, clear of every ledge and brass-strip front). Sprite is two C4-symmetric spin frames (names 96/98) so the rotating chamber can never show it at a wrong angle (INV-HW-004/D-005); it freezes and hides during the rotation tween (the world freeze — main.c gates `ent_update` on `cham_rot`). Rationale: (a) a solid-bouncing floater can drift onto a brass strip and fizzle portal placement mid-puzzle, or sit in a portal hole and be entombed when the portal recalls — the leash makes both states unreachable instead of handled; (b) it's ~60 cycles/frame, invisible to the frame budget; (c) INV-ENG-004 occupancy still applies through the live list, so the placement rule stays consistent. Alternatives: wall-bouncing with raw-solid checks (rejected: the two hazard states above, for no design win yet); player-chasing (deferred — a chase needs the damage system, Phase 3.5). Contact damage lands with death/respawn (Phase 3.5). Revisit if a puzzle needs a drone that interacts with geometry.

**2026-07-12** · Context: Jeremy's first playtest. The GDD's initial scheme (Y fires the CURRENT color, X toggles) read as "can't shoot the second portal" — refiring moved the first one, and the toggle wasn't discoverable mid-play. ## D-012 — Room runtime format: per-8×8-tile attributes; metatiles are an authoring convention; ASCII grid is the authoring source
**2026-07-11** · Context: Phase 1 room loader design (the "metatile semantics" D-011 deferred). Decision: the runtime consumes tmx2snes outputs near-verbatim — the room is a 128×64 grid of SNES map words (tmx2snes `.m16` with `.t16` palette/priority baked in at build time by `tools/roomglue.py`, which also emits the INV-ENG-005 checksum), and collision/material is a per-tileset-tile u16 (`.b16`: low byte collision class, high byte material) looked up through the map word. 16×16 "metatiles" survive as the authoring grid: rooms are authored as a 64×32 ASCII grid (`assets/maps/*.txt`, one char = one metatile = a 2×2 tile quad from `tools/tilesdef.py`), converted to Tiled `.tmj` by `tools/mkmaps.py`, then through tmx2snes (D-011 honored — it stays the sole map converter). Rationale: (a) rendering streams map words straight to VRAM with zero metatile expansion; (b) collision at 8×8 granularity is finer, not coarser, than the planned 16×16 and the query is two array reads; (c) a 64×32-metatile room costs 16.5KB ROM uncompressed — at 512KB with a Phase 5 pressure check, per-room cost is a non-issue; (d) ASCII grids diff cleanly in git and need no GUI overnight — Tiled interchange stays available since `.tmj` is the pipeline's native input (a future Tiled-authored map drops in unchanged). Alternatives: literal metatile-def + grid format (rejected: adds a runtime expansion layer and a custom converter for no measured saving); hand-written `.tmx` in Tiled (deferred to Jeremy's art passes, not a machine workflow).

## D-011 — Map pipeline: tmx2snes adopted; R4 custom-converter fallback NOT triggered
**2026-07-11** · Context: ROADMAP Phase 0 spike. Verified by running tmx2snes 4.5.0 on the pvsneslib likemario map: outputs (`.m16` map, `.b16` tile attributes, `.o16` objects, `.t16` palette/priority) are **byte-identical to the example's committed goldens** (deterministic), and custom tile properties DO carry — each tile's Tiled `attribute` property is a free u16 written verbatim (little-endian) to `.b16`, enough for skyfell's collision byte + material byte; `palette`/`priority` land in `.t16`. Caveats: the converter reads Tiled's **JSON export (`.tmj`)**, not raw `.tmx` (author `.tmx` with an `editorsettings` auto-export, commit both); it's 8×8-tile oriented with named layers — 16×16 metatile semantics are the room loader's job (Phase 1 design). Alternatives: custom Python converter (R4 fallback) — not needed; revisit only if metatile authoring in Tiled proves painful.

## D-010 — Debug block re-pinned to `$7EFF00` (was `$7E1F00`); magic stays `0x51FE`
**2026-07-11** · Context: sibling repo `../prophet` proved during its Phase 0 bring-up that `$7E1F00` is unusable on PVSnesLib 4.5.0 — `crt0_snes.asm` (line 244) sets the stack to `$1FFF` growing **down** through the `$1F00` page, colliding with any block there (`prophet/dbg.asm:10-12`, `prophet/tests/README.md:19`). Skyfell had pinned `$7E1F00` at planning time, before any code existed; no test or offset ever shipped against it, so this is a paper-only move, not an INV-TEST-001 violation. Decision: block lives at `$7E:FF00` (top page of WRAM bank `$7E`), declared `.RAMSECTION ".dbg" BANK 126 SLOT 2 ORGA $FF00 FORCE` — BANK must be 126 (= `$7E`), not 0, because tcc816 addresses the labels with absolute-long (`sta.l`); `FORCE` makes wlalink fail loudly on any collision. Alternatives: another low page (rejected — anything under `$2000` is stack/DP territory and low WRAM is mirrored, inviting aliasing confusion). Jeremy approved 2026-07-11.

## D-005 — Gravity model: world stays fixed; camera/gravity rotate
Context: the marquee mechanic needs a bug-resistant model. Decision: collision/world data never rotates; gravity is a 4-state vector; physics runs in the "gravity frame" (axis remap); rendering applies the rotation so gravity always points screen-down; **chamber enemies are radially symmetric sprites** because OBJ hardware cannot rotate. Alternatives: rotating the tilemap data (rejected — 4× data or expensive transforms, and a whole class of desync bugs); pre-drawn rotated enemy frames (rejected for v1 — VRAM cost; round enemies are charming anyway).

## D-004 — Mode 7 confined to dedicated chamber rooms
Context: could the whole game be Mode 7? Alternatives: full-time Mode 7 Metroidvania (rejected: single BG layer, 8bpp palette pressure, no BG3 HUD, sprite-rotation mismatch everywhere); no Mode 7 at all (rejected: it's the pitch). Chambers get a dedicated VRAM/CGRAM layout behind force-blank door transitions. Fallback if the spin doesn't feel good (Phase 3 gate): instant 90° cuts + tumble animation.

## D-003 — Portals are axis-aligned only (4 orientations)
Alternatives: 45° surfaces (Portal-authentic, but explodes the transform table, collision, and art). 4×4 orientation pairs → one 16-entry LUT (INV-ENG-003). 45° lives in Future Considerations behind a prototype.

## D-002 — Emulator/test target: MesenCE; accuracy spot-checks on ares
Context: verified 2026-07-07 that SourMesen/Mesen2 is archived (at ~2.1.1, mid-2025) and development continues at **github.com/nesdev-org/MesenCE**. Mesen2-line has SNES support, Lua scripting, and a `testrunner` CLI mode (confirmed in source: `UI/Utilities/CommandLineHelper.cs` — `IsTestRunner`, `.lua` args, `timeout=`). Alternatives: bsnes/ares (accuracy gold but no Lua harness), snes9x (weaker debugging). ares is the pre-release accuracy cross-check; real hardware in Phase 7.

## D-001 — Language: C on PVSnesLib 4.5.0, assembly only where profiling demands
Context: verified 2026-07-07 — PVSnesLib 4.5.0 released 2025-12-28, MIT, Windows 64-bit zip, active; bundles 816-tcc toolchain + `816-opt`, `gfx4snes`, `tmx2snes`, `smconv`, `snesbrr`; ships a platformer example (`snes-examples/games/likemario`). Alternatives: pure 65816 asm via ca65/libSFX (maximum control and cool-factor, but 3–5× slower iteration and a much bigger bug surface for a game this systemic); other C kits (none as maintained). Hot paths (collision, OAM build) drop to asm **after** the Mesen profiler fingers them — not speculatively.
